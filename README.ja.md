Maestro Orchestrator は、複数のシミュレーションAIエージェントが安全に協調できるかを検証するためのローカル研究ツールです。

何が起きたかを記録し、出力の一貫性を確認し、危険そうな挙動があれば人間によるレビューのために停止します。

現実のシステムを制御したり、外部へデータを送信したり、自動で修正を適用したり、人間の判断を置き換えたりするものではありません。

研究者・メンテナー向けには、このリポジトリは、明示的な安全チェック、監査ログ、チェックポイント、人間によるレビューを備えた複数エージェントおよびエージェント型ワークフローを研究するためのものです。

このリポジトリは、ローカルシミュレーションと防御的レビュー用ワークフローに焦点を当てています。主に次のような問いを検証するためのものです。

- エージェントワークフローをいつ継続すべきか
- いつ安全に停止すべきか
- いつ人間によるレビューを求めるべきか
- ログと生成物が一貫しているか
- 助言的な検出結果と自動実行が分離されているか
- チェックポイントと監査記録によって予期しない変更を検出できるか

このプロジェクトは本番用の自動化システムではありません。  
法律、医療、金融、規制対応、安全性についての保証を提供するものでもありません。

基本的な設計原則は単純です。

> 挙動が不確実、危険、不整合、または外部に影響する可能性がある場合、ワークフローは自動継続せず、安全に停止するか、人間によるレビューを求めるべきです。

## このプロジェクトは何か

実用上、このプロジェクトは次のような問いを検証するためのものです。

- シミュレーション上のエージェントはタスクに従ったか
- 出力は一貫していたか
- 危険そうな挙動があったときにワークフローは停止したか
- 何が起きたかの記録は残っているか
- 人間によるレビューが必要だったか

このリポジトリの例はローカルシミュレーションです。本番用の自動化ツールではありません。

## 研究者・メンテナー向け

このリポジトリは、複数エージェントのオーケストレーションについて、次の要素を使って研究します。

- 明示的なゲート判断
- 助言専用レビュー
- human-in-the-loop チェックポイント
- 改ざん検出可能な監査ログ
- 結果一貫性チェック
- 不確実時に安全側へ停止する挙動
- 再現可能なローカルシミュレーション
- 助言と実行の明確な分離

実装とテストは、ワークフローの挙動を検査、再現、レビューしやすくすることを目的としています。

## 用語集

- **Agent**: タスクの一部を実行するシミュレーション上の作業者です。
- **Maestro**: タスクを振り分ける調整役です。人間の判断を置き換えるものではありません。
- **Tasukeru**: 防御的なリポジトリレビューに使う助言的レビュー用ワークフローです。
- **HITL**: human-in-the-loop の略です。ワークフローが人間レビューのために停止することを意味します。
- **Advisory-only**: 検出結果を報告するだけで、自動修正、自動コミット、自動push、自動マージを行わないことです。
- **Audit log**: 重要な判断やイベントの記録です。
- **Checkpoint**: シミュレーションを再開するために保存された状態です。
- **Fail-closed**: 挙動が不確実または危険な場合に、継続せず安全に停止するか、レビューを求めることです。
- **External side effect**: ローカルシミュレーションの外に影響する操作です。送信、アップロード、push、削除、デプロイ、現実システムの制御などが含まれます。

## 初めて読む人向けの読み方

このリポジトリを初めて読む場合は、次の順で読むことを推奨します。

1. **安全性と適用範囲** を読む。
2. **責任ある利用と禁止される利用** を読む。
3. **助言専用ポリシー** を読む。
4. **人間によるレビューモデル** を読む。
5. 安全上の境界を理解してから、**現在のシミュレーター系統** を読む。

このリポジトリは研究・教育目的のテストベンチです。  
出力は研究用の成果物であり、本番承認や安全保証ではありません。

挙動が不明な場合は、変更を適用する前に、関連する実装とテストを一緒に確認してください。

## 安全性と適用範囲

このリポジトリは研究用プロトタイプです。

次の用途は想定していません。

- 本番環境での自律的な意思決定
- 教師なしの現実世界制御
- 法律、医療、金融、規制に関する助言
- 実在する個人データや機密運用データの処理
- 完全または普遍的な安全性保証の提示
- 外部への自動送信、アップロード、送付、デプロイ
- 外部副作用に対する人間レビューの迂回

このリポジトリの例は、次のものとして読んでください。

- ローカル研究シミュレーション
- 教育用参考実装
- ガバナンスと安全性のテストベンチ
- 不確実時に安全側へ停止する設計例
- 人間レビュー付きワークフローの設計例
- 監査ログとチェックポイント完全性の実験
- ローカルなバッチ型オーケストレーション例

外部送信、アップロード、push、送付、デプロイ、削除、現実世界制御などの操作は、ブロックするか、人間レビュー付きにする必要があります。

## 責任ある利用と禁止される利用

レビュー用ワークフローとシミュレーター例は、自分が所有するリポジトリ、または明示的な許可を得ているリポジトリの防御的レビューを目的としています。

このリポジトリまたはツールを、次の目的で使用しないでください。

- 無許可の脆弱性探索
- 攻撃的な偵察
- 第三者のシステムやコードの悪用
- 認証情報、トークン、秘密情報の収集
- 許可のないリポジトリやシステムのスキャン
- 実在する対象に対する攻撃手順の生成または検証
- アクセス制御、レート制限、セキュリティ境界の迂回
- 責任ある開示を経ない機微な検出結果の公開

検出結果は、安全性、信頼性、追跡可能性、ガバナンスの改善に使用してください。第三者またはオープンソースのコードをレビューする場合は、そのプロジェクトのセキュリティポリシーと責任ある開示プロセスに従ってください。

## 助言的レビュー用ワークフロー

Tasukeru Analysis は、このリポジトリにおける防御的レビュー用の助言的ワークフローです。

これは、複数エージェントやエージェント型ワークフローが複雑化する中で、その挙動を検査、説明、監査しやすくすることを目的としています。

主に次のようなレビュー支援の問いを扱います。

- 検出結果が適切なレビュー水準に分類されているか
- プロセスログと生成物が一貫しているか
- 不確実または危険なケースが人間レビューへ戻されているか
- 公開PRコメントが過度に詳細なセキュリティ情報を含んでいないか
- ワークフローが自動修正、自動コミット、自動push、自動マージを行わない助言専用のままか

このワークフローは、詳細な悪用手順を公開せず、自律的な強制システムとしても動作しません。詳細な検出結果は、人間レビュー用のワークフロー成果物やログに残します。

実行されるチェックには、次のようなものがあります。

- Ruff
- Bandit
- pip-audit
- リポジトリ固有のロジックチェック
- ドキュメントレビュー
- 人間レビュー分類

このワークフローは、すべての助言的検出結果を自動的にブロッキング失敗として扱うのではなく、保守性、安全レビュー、開発者の使いやすさを改善するためのものです。

警告を隠すことは目的ではありません。代わりに、積極的な修正候補、レビューのみ必要な項目、過去バージョンの警告、ノイズ候補を分離し、保守者が優先すべき項目に集中できるようにします。

## 助言専用ポリシー

助言的レビュー用ワークフローは、次のことを行いません。

- ブランチの自動作成
- Pull Request の自動作成
- 自動コミット
- 自動push
- 自動修正の適用
- Pull Request の自動マージ

人間レビューが必要な検出結果のみ、PRコメントを生成する場合があります。

その他の検出結果は、GitHub Step Summary とワークフロー成果物に収集され、人間レビュー用の材料として扱われます。

このワークフローは、人間のレビュー負荷を減らすためのものであり、人間の判断を置き換えるものではありません。

## 人間可読性ポリシー

このプロジェクトは、人間が安全に読めて、レビューできて、保守できるコードを優先します。

AIや自動ツールにとって処理しやすいという理由だけで、その変更をよい修正として扱うべきではありません。人間の可読性、レビュー容易性、保守性を下げる可能性がある変更は、人間レビューへ戻すべきです。

## セキュリティレビュー出力ポリシー

セキュリティ関連の出力は、防御的レビューと安全な修正方針に限定するべきです。

公開PRコメントには、次の内容を含めないでください。

- exploit payload
- 攻撃コマンド
- 攻撃的な再現手順
- 段階的な悪用手順
- 第三者対象への攻撃詳細

セキュリティ関連の検出結果は、次の点に焦点を当てるべきです。

- 影響を受けるファイルと行
- レビューが必要な防御上の理由
- 想定される安全上の影響
- 安全な修正方向
- 人間レビューが必要かどうか

## 人間によるレビューモデル

このリポジトリでは、実行可能な問題と想定内のノイズを分離するためにレビュー水準を使用します。

現在のレビュー水準は次の通りです。

- `HITL_REQUIRED`: マージまたは次の操作の前に人間レビューが必要
- `FIX_RECOMMENDED`: 具体的な修正が適切である可能性が高い
- `REVIEW_RECOMMENDED`: 判断前に文脈の確認が必要
- `NOISE_CANDIDATE`: テスト、デモ、研究シミュレーションでは許容可能である可能性が高い
- `INFO_ONLY`: 過去バージョンの警告を含む情報目的の項目

この分類は原則として助言です。保守者が、修正するか、文書化するか、抑制するか、許容するかを判断するための材料です。

## アクティブレビューキュー

メインのレビューキューは、コンパクトに保つことを意図しています。

直ちに注意が必要な検出結果は、次のいずれかとして表示されるべきです。

- `HITL_REQUIRED`
- `FIX_RECOMMENDED`
- `REVIEW_RECOMMENDED`

これらの件数がゼロの場合、ワークフローが現在優先的な人間修理またはレビューを推奨しているアクティブな助言項目はありません。

これは、リポジトリに警告が存在しないことを意味しません。残りの警告が、ノイズ候補または過去・情報目的の記録として分類されていることを意味します。

## 過去バージョンの警告

過去バージョンの警告は、完全には除外しません。

助言的ワークフローは、置き換え済みまたは過去のシミュレーターファイルからの検出結果を、監査上有用だが現在の修正対象ではない場合に `INFO_ONLY` として可視化します。

これにより、現在人間の注意が必要な検出結果にアクティブレビューキューを集中させながら、次の目的で保持されている古いバージョンファイルの可視性も維持できます。

- 再現性
- 回帰比較
- 履歴参照
- バージョン付き実験
- 設計比較

そのファイルが再びアクティブな入口になる場合は、過去の警告も再確認してください。

## 典型的な分類例

例:

### subprocess の実行

通常は `HITL_REQUIRED` です。

subprocess の実行は外部副作用を生む可能性があるため、レビューが必要です。

### subprocess の import

通常は `REVIEW_RECOMMENDED` です。

subprocess を import すること自体は外部副作用ではありません。

### テスト内の assert 使用

通常は `NOISE_CANDIDATE` です。

pytest のテストでは一般的に `assert` が使用されます。

### アクティブな実行時コード内の assert 使用

通常は `FIX_RECOMMENDED` です。

実行時コードの assert 文は、Python の最適化フラグによって削除される可能性があります。

### 置き換え済みの過去シミュレーター内の assert 使用

通常は `INFO_ONLY` です。

完全に除外せず、過去バージョンの警告として可視化します。そのファイルが再びアクティブな入口にならない限り、直ちに修正する必要はありません。

### テスト内の一時パス使用

`tmp_path` のような隔離された pytest fixture を使用している場合、通常は `NOISE_CANDIDATE` です。

ハードコードされた共有パスを使用している場合は、レビューするべきです。

### アクティブなシミュレーション内の疑似乱数使用

通常は `REVIEW_RECOMMENDED` です。

決定的なシミュレーション用の乱数は、秘密情報、トークン、認証、認可、暗号用途に使用されていない場合は許容できることがあります。

許容する場合は、その乱数がシミュレーション専用であることをコード上で文書化するべきです。

### 置き換え済みの過去シミュレーター内の疑似乱数使用

通常は `INFO_ONLY` です。

過去バージョンの警告として可視化します。セキュリティ上重要な挙動やアクティブな実行時挙動に再利用されない限り、直ちに修正する必要はありません。

### 依存関係監査ツールによる脆弱性検出

通常は `FIX_RECOMMENDED` です。

## 証拠、説明、予測

助言的ワークフローは、検出結果に構造化された判断支援メタデータを付与する場合があります。

これには次の情報が含まれる場合があります。

- `evidence`: ソースツール、ルールID、ファイル、行、スニペット、成果物参照
- `explanation`: なぜその検出結果が重要か、なぜその修正方向が提案されるか
- `fix_prediction`: 提案変更によって期待される安全性または保守性上の効果
- `fix_verification`: 候補検証用のプレースホルダーまたは結果データ

`fix_prediction` は保証ではありません。レビュー支援情報です。

`fix_verification` が有効な場合も、助言専用でなければなりません。候補チェックは一時ファイルまたは隔離されたファイルを使用し、人間が明示的に手動適用を選ばない限り、リポジトリファイルを変更してはいけません。

## 出力成果物

助言的ワークフローは、次のようなレビュー成果物を生成する場合があります。

- 助言サマリー
- 人間レビュー用レポート
- 機械可読なレビュー用データ
- 通知状態記録
- PRコメント案
- ドキュメント提案ドラフト
- 信頼度レポート
- 結果一貫性レポート
- 依存関係一貫性レポート

これらの成果物はレビュー材料です。リポジトリファイルを変更せず、自動適用されてはいけません。

人間可読なレビュー用レポートには、次の情報が含まれる場合があります。

- ファイルパス
- 行番号
- 検出コード
- 問題概要
- レビュー水準
- 推奨アクション

機械可読なレポートは、同じ判断支援データを構造化された形式で提供します。

PRコメント案は、人間レビューが必要であり、PR通知の条件を満たす検出結果にのみ使用されます。

ドキュメント提案ドラフトには、最小差分案、全文ドラフト、ハッシュ、証拠、レビュー用チェックリストが含まれる場合があります。これらはリポジトリファイルを変更せず、自動適用されてはいけません。

信頼度スコアはレビュー支援用メタデータにすぎません。検出結果を無視して安全であることを示すものではなく、自動修正、自動PR作成、自動コミット、自動push、自動マージを有効化してはいけません。

結果一貫性レポートは、プロセスログと生成された結果成果物が一致しているかを記録します。

依存関係一貫性レポートは、検出結果、影響を受ける構造、影響分類、レビュー水準、生成物が一貫しているかを記録します。

これらの成果物はすべて助言専用です。検出結果を変更せず、分類を変更せず、修正を適用せず、コミット、push、Pull Request のマージを自動実行しません。

## 助言的な挙動

助言的ワークフローは、保守者が次の問いに答えられるよう支援するべきです。

- どの検出結果がノイズ候補か
- どの検出結果をレビューすべきか
- どのファイルに注意が必要か
- なぜその検出結果が関係するのか
- 推奨される修正は何か
- マージ前に人間レビューが必要か
- それはアクティブな修正候補か、過去バージョンの警告か

このワークフローは人間レビューを置き換えません。より安全なリポジトリ保守のためのトリアージと文書化支援です。

## リポジトリの目的

このリポジトリの主目的は、エージェント型ワークフローを明示的なチェック、再現可能なログ、中断点、人間レビューによってどのように制御できるかを探ることです。

このプロジェクトは次の点を重視します。

- 不確実時に安全側へ停止する挙動
- 危険な操作前の人間レビュー
- 固定された安全チェック順序
- 安定した理由コード
- チェックポイントと再開
- 改ざん検出可能な監査記録
- 成果物の完全性検証
- 再現可能なシミュレーション出力
- 助言と実行の明確な分離

このリポジトリは、完全なAI安全性を提供すると主張するものではありません。オーケストレーション挙動を研究するための研究・教育用テストベンチです。

## 現在のシミュレーター系統

このリポジトリには、複数のローカル専用シミュレーター系統があります。それぞれ異なるオーケストレーションまたはレビュー上の問題に焦点を当てています。

### 1. ゲートベースのメディエーター・シミュレーター

このシミュレーターは、単純な流れを検証します。

```text
Agent → Mediator → Orchestrator
```

主な特徴:

- エージェントがタスク入力を正規化する
- メディエーターは助言のみを行う
- メディエーターは実行権限を持たない
- オーケストレーターは固定順序でチェックを評価する
- 曖昧または相対的な要求は人間レビューへ送られる
- 高リスクなケースはブロックされる場合がある
- 生テキストは永続化しないべきである
- 監査ログでは直接的な個人データ漏えいを避けるべきである

標準的なチェック順序:

```text
Meaning → Consistency → Ambiguity Review → External-Action Safety → Dispatch
```

このシミュレーターは、次の確認に有用です。

- 固定されたチェック順序
- 人間レビューへの遷移
- ブロックされる挙動とブロックされない挙動
- メディエーターとオーケストレーターの分離
- 理由コードの期待値
- 軽量な複数タスクのオーケストレーション挙動

### 2. ドキュメントタスク・チェックポイント・シミュレーター

このシミュレーターは、より強い永続化と完全性制御を備えたドキュメントタスクのオーケストレーションを検証します。

主な特徴:

- タスク契約チェック
- 大まかなトークン推定
- Word / Excel / PowerPoint 風タスクシミュレーション
- タスクごとのチェックポイントと再開
- 監査ログのハッシュチェーン
- 保護されたチェックポイントファイル
- 成果物の完全性記録
- 個人データに配慮した監査、チェックポイント、成果物の書き込み
- 改ざん検出
- CLIベースのローカルシミュレーション
- 外部への自動送信なし

重要な実装上の注意:

このシミュレーターは、ドキュメントタスク結果を表すテキストベースの成果物を出力します。別途ドキュメント生成ライブラリを接続し、テストしない限り、完全に整形された Microsoft Office 文書を生成すると主張するものではありません。

ハッシュと HMAC は改ざん検出を提供します。物理的な書き込み防止を提供するものではありません。実運用に近い環境では、HMACキーを環境変数または保護されたキーファイルから供給し、リポジトリへコミットしてはいけません。

### 3. 緊急契約ゲート・オーケストレーション・シミュレーター

このシミュレーターは、緊急契約シナリオとゲートベースのオーケストレーションフローを組み合わせたものです。

これは小規模な統合概念実証であり、本番の契約システム、法律ツール、信号制御システムではありません。

主な特徴:

- 緊急契約シナリオ
- 固定された安全チェック順序
- 優先順位が曖昧なケースのレビュー挙動
- 証拠検証と捏造証拠の検出
- 人間承認チェックポイント
- 管理者最終確認チェックポイント
- 高リスク条件に対するブロック停止の不変条件
- シミュレーション用契約ドラフト成果物の生成
- 改ざん検出可能な監査行
- 現実世界の信号制御なし
- 法的効力なし
- 外部送信、アップロード、送付、API呼び出し、デプロイなし

標準的な統合フロー:

```text
Meaning → Consistency → Ambiguity Review → Evidence Check → Human Authorization
→ Draft → Safety Review → Draft Review → External-Action Safety
→ Admin Finalize → Dispatch
```

このシミュレーターは、次の確認に有用です。

- 緊急契約シナリオの扱い
- 相対的な優先順位評価
- 捏造証拠による一時停止挙動
- 現実世界制御のブロック
- ユーザーおよび管理者による停止経路
- シミュレーション成果物のみの dispatch
- 監査ログ検証
- 正常系と異常系にまたがる安全不変条件

### 4. インフラ・ライフライン調停シミュレーション

このシミュレーターは、3つのインフラエージェントを含む制約付き資源配分シナリオをモデル化します。

- electricity
- water
- gas

これは、障害時の制約された資源のもとで、メディエーターが提案専用の配分案を作る方法を研究するための、ローカルかつseed再現可能な研究シミュレーションです。

主な特徴:

- 通常時必要資源を持つ3つのインフラエージェント
- 障害モードでの総資源制約
- 最小保証
- 優先度重み
- 生命リスクスコア
- 不足率を考慮した配分
- シミュレーション上の人間判断
- JSON と stdout 出力
- 外部APIアクセスなし
- 実インフラ制御なし
- 自動復旧なし
- 自動停止または切断なし

メディエーターは配分案のみを作成します。実インフラを制御したり、現実世界の復旧操作を行ったりしません。

シミュレーション上の人間オペレーターは、次のいずれかを返す場合があります。

- `APPROVE`
- `REJECT`
- `REDEFINE`
- `REQUEST_ALTERNATIVES`

このシミュレーターは、次の確認に有用です。

- 制約付き資源配分挙動
- 提案専用の調停
- seed による再現性
- 人間レビュー分岐
- 安全境界フラグ
- JSON出力挙動
- 配分合計が利用可能資源の上限内に収まること

### 8. Officeタスク調停シミュレーション

このシミュレーターは、Word / Excel / PowerPoint 風の合成成果物を使う、ローカル専用の Office タスク調停フローをモデル化します。

生成された文書、表計算、プレゼンテーション出力が、元のユーザータスク指示と一致しているかを確認します。診断支援機構はログを読み、異常を検出し、リスク材料のみを出力します。メディエーターはマスク済みメタデータパケットのみを受け取り、元タスクと生成出力の差異を調停します。コーディネーターは自分で判断しません。ユーザーが選択したエージェントにのみタスクを配布し、スコアが閾値を超えた場合のみ人間レビューを起動し、明示的に選択されたユーザー操作のみを実行します。

シミュレーションフロー:

```text
ユーザーが対象エージェントを選択
↓
コーディネーターがユーザー選択済みエージェントにのみタスクを配布
↓
Word / Excel / PowerPoint 風の合成成果物を生成
↓
診断支援機構が内部ログを読み、マスク済みメタデータパケットを作成
↓
メディエーターがマスク済みメタデータのみを受け取り、元タスクスナップショットと比較
↓
メディエーターが衝突スコアを計算
↓
リスク推定が助言メタデータとして付与される
↓
score == 0.8 の場合、結果は warning / draft review のまま
↓
score > 0.8 の場合、実行は人間レビューのために一時停止
↓
ユーザーが次の操作を選択
↓
コーディネーターが明示的に選択されたユーザー操作のみを実行
```

このシミュレーターは実際の Office ファイルを生成しません。Word / Excel / PowerPoint 風の合成レコードを使って、一貫性、マスキング、閾値挙動、人間レビューへのルーティング、ドラフト専用の修正伝播をテストします。

主な特徴:

- 固定された Word / Excel / PowerPoint 合成タスクセット
- ユーザー選択済みエージェントへの dispatch
- 元タスクスナップショットとハッシュ固定されたタスク基準
- 診断ログ分析とマスク済みメタデータハンドオフ
- メディエーターへの生ログ引き渡しなし
- Office出力の一貫性チェック
- 利益、数式、グラフ、結論の不一致検出
- 個人データと機密シグナルのマスキング
- 閾値ポリシー: `score == 0.8` は warning / draft review のみ
- 閾値ポリシー: `score > 0.8` は人間レビューを起動
- ユーザー範囲のドラフト修正提案のみ
- 関連エージェントはドラフト修正提案のみを作成
- コーディネーターに自律的判断権限なし
- 監査ログ検証
- 結果一貫性検証
- 必須 Word / Excel / PowerPoint 成果物の完全性チェック
- 必須成果物の欠落検出
- ゼロ値を有効値として扱う
- 安全バッファメタデータが明示的かつレビュー可能
- 内部生エージェントログはデフォルトでは保存せず、明示的なシミュレーション専用オプトインが必要
- 実 Office 文書生成なし
- 外部APIアクセスなし
- 実プロセス制御なし
- 自動修正、コミット、push、マージなし

このシミュレーターは、次の確認に有用です。

- Word / Excel / PowerPoint 風出力が一貫しているか
- Excel の数式結果と PowerPoint のグラフ値が衝突していないか
- Word本文、表計算値、プレゼン要約が食い違っていないか
- 元のユーザー指示が比較基準として維持されているか
- 個人データと機密シグナルが調停前にマスクされているか
- メディエーターがマスク済みメタデータのみを使用しているか
- `score == 0.8` が人間レビューを起動しないか
- `score > 0.8` が人間レビューを起動するか
- ユーザー対象の修正プロンプトがドラフト提案のみを生成するか
- コーディネーターが自律判断を避け、明示的に選択されたユーザー操作のみを実行するか

### 9. 信頼度からリスクへの自動化ドラフト拡張

このドラフト拡張は、Officeタスク調停系統に、自動化開始診断と自動化継続診断を追加します。

主な特徴:

- 信頼度スコアに基づく自動化開始診断
- `trust_score == 0.9` は自動化対象外
- `trust_score > 0.9` はユーザー承認後に自動化候補になり得る
- 自動化開始後は自動化リスクスコアに基づく継続診断へ切り替える
- `automation_risk_score < 0.1` は自動化継続
- `automation_risk_score >= 0.1` は安全側への一時停止を起動
- コーディネーターは一時的な自動化停止を中継するだけ
- 自動化再開にはユーザーレビューが必要
- 診断支援機構は診断専用であり、エージェントへ直接命令しない
- 自動修正、コミット、push、マージなし

この拡張は、次の確認に有用です。

- 自動化開始が `trust_score > 0.9` によって制限されるか
- `trust_score == 0.9` が対象外として扱われるか
- 自動化が有効になる前にユーザー承認が必要か
- 自動化継続が信頼度スコアからリスクスコアへ切り替わるか
- `automation_risk_score == 0.1` が安全側の一時停止になるか
- 停止した自動化の再開にユーザーレビューが必要か
- 自動修正、コミット、push、マージが無効のままか

### 10. ソース根拠付き Office オーケストレーション・シミュレーション

このシミュレーターは、Officeタスク調停系統を、成果物一貫性チェックからソース根拠付き複数エージェント・オーケストレーションへ拡張します。

主な追加点:

- primary / secondary / tertiary の合成ソースエージェント
- 生ソースログではなくソース証拠パケットを使用
- ソースと成果物の一貫性チェック
- ソースパケットに基づく Office ドラフト成果物
- ソース衝突と成果物矛盾に対するメディエーター調停
- 診断用ソース/成果物パケットの境界チェック
- 個人データと機密シグナルのマスキング
- 根拠のない主張の検出

このシミュレーターはローカル専用かつ合成データ専用です。実際のソースを取得せず、実 Office ファイルを生成せず、外部APIを呼び出さず、実プロセスを制御せず、自動修正、コミット、push、マージを行いません。

このシミュレーターは、次の確認に有用です。

- 監査ログと生成物が一貫しているか

### 10.1 Mediator / Maestro ゲート・ハードニング DRAFT

このDRAFT系統は、ソース根拠付き Office オーケストレーション・シミュレーションを拡張し、Mediator から Maestro への明示的なレビュールーティングと、スコア判定における fail-closed 処理を追加するものです。

ファイル:

- `agent_source_grounded_office_orchestration_sim_v1_0_0_refactored_integrated_draft.py`
- `agent_source_grounded_office_orchestration_sim_v1_0_1_fail_closed_hardening_draft.py`
- `tests/test_agent_source_grounded_office_orchestration_sim_v1_0_1_fail_closed_hardening_contract.py`

`v1.0.0` ファイルは比較用の baseline として保持します。  
`v1.0.1` ファイルは、baseline で検証済みの数値境界の挙動を維持したまま、無効な閾値スコア入力を fail-closed に処理する機能を追加します。

Mediator / Maestro のゲート判断:

- `HITL_REVIEW_READY`: 結果を人間のレビュー担当者へ候補として提示できます。ただし、自動承認や外部への自動反映は行いません。
- `PAUSE_FOR_HITL`: 責任点が満たされない場合、残余適合値の閾値を満たさない場合、または閾値入力を安全に評価できない場合に、ワークフローを一時停止します。
- `ROUTED_TO_PRECHECK`: 重大条件または構造的不変条件違反が検出された場合、通常の候補比較へ進まず、前段確認へ移送します。

fail-closed な閾値処理:

```text
無効なスコア入力
→ PAUSE_FOR_HITL
```

対応する契約テストでは、次を確認します。

- 無効なスコア入力が `PAUSE_FOR_HITL` に安全側で閉じること
- baseline の数値閾値挙動が維持されていること
- 責任点の失敗がスコアによって救済されないこと
- 重大条件および構造的不変条件違反が `ROUTED_TO_PRECHECK` へ移送されること
- 候補提示と次段階への承認が分離されていること
- ユーザー承認によって `PAUSE_FOR_HITL` または `ROUTED_TO_PRECHECK` を迂回できないこと
- Maestro が自律的な最終判断または外部操作を行わないこと
- Tasukeru、Mediator、Maestro が不正な `sealed=True` 状態を発行しないこと
- `runs=100` および `runs=1000` の決定論的な分布確認

これは、ローカル専用の研究・検証用DRAFTです。  
自動採用、外部への自動作用、自動修正、自動コミット、自動push、自動マージを許可するものではありません。

## バッチ実行と再開

このリポジトリには、バッチ型オーケストレーションの例も含まれています。

バッチ実行とは、複数のドキュメント関連タスクを1つのオーケストレーション実行として評価しつつ、チェック判断、監査記録、中断点を保持することを意味します。

### メディエーターベースのバッチフロー

メディエーターベースのシミュレーターは、1回の実行で複数タスクを受け取ることができます。

利用例:

- 複数の表計算またはスライドタスクをまとめて評価する
- タスク単位のチェック結果を比較する
- 複数タスクにまたがる人間レビュー挙動を確認する
- オーケストレーション前にメディエーター助言を確認する
- 各タスクが固定チェック順序を通ることを確認する

この系統は、次に焦点を当てる場合に有用です。

- 複数タスクのチェック挙動
- メディエーター分離
- 曖昧性レビューと人間レビューへの遷移
- 理由コードの安定性
- 軽量なオーケストレーションテスト

タスク列の例:

- T1: 表計算タスク
- T2: プレゼンテーションタスク
- T3: 人間レビューが必要な曖昧タスク

期待される挙動:

- 各タスクはエージェントによって正規化される
- 各タスクはメディエーター助言を受け取る
- 各タスクはオーケストレーターによって評価される
- 一時停止したタスクは、高リスク安全チェックがブロックしない限り、最終停止として扱われない
- 最終判断が継続の場合のみ dispatch される

### ドキュメントタスク・バッチフロー

ドキュメントタスク・チェックポイント・シミュレーターは、固定されたドキュメントタスク列を使用します。

```text
Word → Excel → PowerPoint
```

この系統は、次に焦点を当てる場合に有用です。

- バッチタスク実行
- タスクごとのチェックポイント
- 中断と再開
- 成果物の完全性記録
- 保護されたチェックポイント
- 監査ログのハッシュチェーン
- 改ざん検出

タスクが中断された場合、シミュレーターは次の情報を記録します。

- 失敗したタスクID
- 失敗したチェック
- 理由コード
- チェックポイントパス
- 再開要件
- 必要な場合の人間確認要件

後続の実行では、必要に応じて人間確認を経たうえで、チェックポイントから再開できます。

## バッチスクリプト

バッチスクリプトは、ローカルシミュレーション実行のための便利なラッパーとして使用できます。

推奨されるスクリプト例:

- ローカルデモ実行
- チェックポイント再開実行
- 改ざん検出チェック実行

これらのスクリプトは、ローカル開発者用ユーティリティとしてのみ扱ってください。

次のことを行ってはいけません。

- 本番用HMACキーを埋め込む
- 成果物を自動的にアップロードまたは提出する
- 人間確認を迂回する
- ファイルを自動削除する
- 実在する個人データまたは機密データを処理する
- ライセンスや安全性の意味を変更する
- テストまたは安全チェックの不変条件を弱める

ローカルデモでは、シミュレーターが明示的に対応している場合のみデモキーモードを使用できます。実運用に近い実行では、環境変数または保護されたキーファイルから HMAC キーを供給してください。

### バッチスクリプトの役割例

#### ローカルデモスクリプト

目的:

- ローカルデモを実行する
- 安全なサンプル入力を使用する
- 監査ログとシミュレーション成果物をローカル出力ディレクトリに書き込む
- 外部送信を避ける

#### 再開スクリプト

目的:

- チェックポイントから再開する
- 明示的な再開確認を要求する
- 継続前に完了済み成果物を検証する
- チェックポイント完全性を維持する

#### 改ざんチェックスクリプト

目的:

- 監査ログ、チェックポイント、成果物の完全性挙動を検証する
- 改ざん検出を実演する
- 完全性検証に失敗した場合は人間レビューのために一時停止する

## 推奨される読み順

ゲート挙動とメディエーター分離を確認する場合:

- メディエーターベースのゲート・シミュレーターを読む
- 対応するテストを読む

この経路は、固定チェック順序、人間レビューへの遷移、ブロックされる挙動とブロックされない挙動、メディエーター分離、軽量な複数タスク・オーケストレーション挙動を学ぶのに有用です。

チェックポイント、再開、成果物完全性、ハッシュチェーン挙動を確認する場合:

- ドキュメントタスク・チェックポイント・シミュレーターを読む
- 対応するテストを読む

この経路は、バッチ実行、チェックポイント、成果物検証、改ざん検出、再開挙動を学ぶのに有用です。

緊急契約挙動を確認する場合:

- 緊急契約ゲート・オーケストレーション・シミュレーターを読む
- 対応するテストを読む

この経路は、具体的な緊急契約シナリオを、固定チェック、人間レビューチェックポイント、監査検証、シミュレーション成果物 dispatch によって、現実世界制御や法的効力なしに評価する方法を学ぶのに有用です。

エージェントインシデント調停挙動を確認する場合:

- エージェントインシデント調停シミュレーターを読む
- 対応するテストを読む

この経路は、診断支援機構がシミュレーション上のログ/出力不一致を検出し、メディエーターへエスカレーションし、人間レビューを起動し、ユーザー指示を記録し、外部副作用なしに出力一貫性を検証する方法を学ぶのに有用です。

コード異常ハンドオフ挙動を確認する場合:

- コード異常ハンドオフ・シミュレーターを読む
- 対応するテストを読む

この経路は、メタデータのみのコード契約異常が検出され、ユーザー管理コーディネーターへエスカレーションされ、人間レビューを受け、安全なチェックポイント・ハンドオフで処理される方法を、外部副作用なしに学ぶのに有用です。

Officeタスク調停挙動を確認する場合:

- Officeタスク調停シミュレーターを読む
- 対応するテストを読む

この経路は、Word / Excel / PowerPoint 風の一貫性チェック、マスク済みメタデータハンドオフ、閾値ベースの人間レビュー挙動、ユーザー対象のドラフト修正が自動適用なしに行われる流れを学ぶのに有用です。

ソース根拠付き Office オーケストレーション挙動を確認する場合:

- ソース根拠付き Office オーケストレーション・シミュレーターを読む
- 対応するテストを読む

この経路は、ソースエージェント、ソース証拠パケット、Officeドラフト成果物、メディエーター調停、診断パケット処理、人間レビュー中継、ループ上限制御が、外部副作用なしにどのように連携するかを学ぶのに有用です。

挙動検証では、常に実装と対応するテストを一緒に読んでください。

特に次の項目では重要です。

- 安全チェックの不変条件

- 人間レビューへの遷移
- ブロックされる挙動とブロックされない挙動
- チェックポイントと再開

- 再現性チェック
- ベンチマーク期待値
- バッチ実行挙動

このリポジトリでは、テストがシミュレーターとオーケストレーションロジックの期待挙動を定義していることがよくあります。

挙動を確認する場合は、実装と対応するテストを一緒に読んでください。

テストでは、次の点を検証する場合があります。

- 固定されたチェック順序
- 不確実時に安全側へ停止する挙動
- 人間レビューへのエスカレーション
- 曖昧性レビュー挙動
- ブロック制約
- チェックポイント復旧
- 監査ログ完全性
- 成果物ハッシュ検証
- 理由コードの安定性
- 再現性の期待値
- バッチタスク挙動
- CLI挙動
- 緊急契約シナリオフロー
- 捏造証拠の扱い
- 現実世界制御のブロック
- ユーザー/管理者による拒否経路
- 改ざん検出

新しいバージョン番号が、常に主要な推奨入口であることを意味するわけではありません。一部のファイルは、履歴比較、再現性、バージョン付き実験のために保持されています。

## ゲートと判断モデル

このリポジトリは、オーケストレーション挙動を追跡可能にするため、明示的なゲート判断を使用します。

よく使われる判断には次のものがあります。

- `RUN`
- `PAUSE_FOR_HITL`
- `STOPPED`

一般的な解釈:

- `RUN`: シミュレーターは次のチェックまたは dispatch ステップへ進める
- `PAUSE_FOR_HITL`: シミュレーターは一時停止し、人間レビューを待つべきである
- `STOPPED`: シミュレーターはブロック条件に到達した

曖昧または相対的な要求は、安全だと暗黙に扱わず、人間レビューのために一時停止するべきです。

外部副作用、ポリシー違反、危険な操作などの高リスク条件は、シミュレーター契約に応じてブロックするか、人間レビューへ送るべきです。

## 監査と完全性モデル

監査と完全性の挙動は、シミュレーター系統ごとに異なります。

メディエーターベースのシミュレーターは、軽量な監査イベントと安全なコンテキストログに焦点を当てます。

ドキュメントタスク・チェックポイント・シミュレーターは、より強い完全性制御を追加します。

- 監査ログのハッシュチェーン
- 行単位の HMAC
- チェックポイント HMAC
- 成果物 SHA-256
- 成果物 HMAC
- 再開時の完了済み成果物検証
- 改ざん検出

これらの仕組みは、変更を検出可能にするためのものです。ローカルの操作者がディスク上のファイルを書き換えること自体を防止するものではありません。

## チェックポイントと再開モデル

チェックポイントベースのシミュレーターは、中断された実行を再開できるように設計されています。

チェックポイントには、次の情報が記録される場合があります。

- run ID
- 現在のタスクID
- 失敗したタスクID
- 失敗したチェック
- 理由コード
- タスク状態
- 成果物パス
- 成果物ハッシュ
- 再開が許可されているか
- 再開前に人間レビューが必要か

再開時には、シミュレーターが続行する前に完了済み成果物を検証するべきです。検証に失敗した場合、黙って継続せず、人間レビューのために一時停止するべきです。

## 成果物モデル

シミュレーターの成果物出力は、研究成果物として扱ってください。

シミュレーター系統によって、出力には次のものが含まれる場合があります。

- 成果物プレビュー
- テキストベースのドキュメントタスク出力
- 監査 JSONL ファイル
- チェックポイント JSON ファイル
- 完全性メタデータ
- サマリー記録

実際のドキュメント生成コードが追加され、テストされていない限り、シミュレーション出力を完全な本番 Office 文書として説明しないでください。

## 外部副作用

外部副作用には、次のような操作が含まれます。

- メール送信
- ファイルのアップロード
- 成果物の提出
- 変更の push
- ファイル削除
- 外部API呼び出し
- ライセンス上の意味変更

これらの操作は、シミュレーター契約に応じて、ブロック、禁止、または人間レビュー付きにする必要があります。

スクリプトやシミュレーターが、外部提出を黙って実行してはいけません。
