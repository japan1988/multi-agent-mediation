Maestro Orchestrator は、複数のシミュレートされたAIエージェントが安全に協働できるかを検証するための、ローカル研究用ツールです。

何が起きたかを記録し、出力が一貫しているかを確認し、リスクがあるように見える場合は人間レビューのために一時停止します。

このプロジェクトは、実システムを制御せず、データを外部送信せず、修正を自動適用せず、人間の判断を置き換えません。

研究者およびメンテナ向けに、このリポジトリは、明示的な安全チェック、監査ログ、チェックポイント、人間レビューを備えたマルチエージェントおよびエージェント型ワークフローを探索します。

このリポジトリは、ローカルシミュレーションと防御的レビュー・ワークフローに焦点を当てています。次のような問いを検証するために設計されています。

- エージェント・ワークフローをいつ継続すべきか
- いつ安全に停止すべきか
- いつ人間レビューを求めるべきか
- ログと生成出力の一貫性が保たれているか
- 助言的な検出結果が自動実行から分離されているか
- チェックポイントと監査記録が予期しない変更を検出できるか

このプロジェクトは本番用の自動化システムではありません。  
法務、医療、金融、規制、安全性に関する保証は提供しません。

基本設計原則は単純です。

> 振る舞いが不確実、危険、不整合、または外部に影響する場合、ワークフローは自動継続するのではなく、安全に停止するか、人間レビューを求めるべきです。

## このプロジェクトとは

実務上、このプロジェクトは次のような問いを検証するために役立ちます。

- シミュレートされたエージェントはタスクに従ったか
- 出力の一貫性は保たれたか
- リスクがありそうな場合にワークフローは停止したか
- 何が起きたかの記録はあるか
- 人間によるレビューが必要だったか

例はローカルシミュレーションです。本番用の自動化ツールではありません。

## 研究者とメンテナ向け

このリポジトリは、次の要素を備えたマルチエージェント・オーケストレーションを探索します。

- 明示的なゲート判断
- 助言のみのレビュー
- human-in-the-loop チェックポイント
- 改ざん検知可能な監査ログ
- 結果一貫性チェック
- fail-closed 動作
- 再現可能なローカルシミュレーション
- 助言と実行の明確な分離

実装とテストは、ワークフローの振る舞いをより検査しやすく、再現しやすく、レビューしやすくすることを目的としています。

## 用語集

- **Agent**: タスクの一部を実行するシミュレートされた作業者。
- **Maestro**: タスクをルーティングする調整役。人間の判断を置き換えません。
- **Tasukeru**: 防御的なリポジトリレビューに使われる助言的レビュー・ワークフロー。
- **HITL**: human-in-the-loop。ワークフローが人間レビューのために一時停止すること。
- **Advisory-only**: ツールは検出結果を報告するが、自動修正、コミット、プッシュ、マージはしないこと。
- **Audit log**: 重要な判断やイベントの記録。
- **Checkpoint**: シミュレーションを再開するために保存される状態。
- **Fail-closed**: 振る舞いが不確実または危険な場合に、継続するのではなく安全に停止するかレビューを求めること。
- **External side effect**: 送信、アップロード、プッシュ、削除、デプロイ、実システム制御など、ローカルシミュレーション外部に対する作用。

## 初学者向けの読み方

このリポジトリを初めて読む場合は、次の順序で始めてください。

1. **Safety and scope** を読む。
2. **Responsible use and prohibited use** を読む。
3. **Advisory-only policy** を読む。
4. **Human review model** を読む。
5. 安全境界を理解した後に **Current simulator lines** を読む。

このリポジトリは研究・教育用のテストベンチです。  
出力は研究成果物であり、本番承認や安全保証ではありません。

振る舞いが不明確な場合は、変更を適用する前に、関連する実装とテストを合わせて読んでください。

## 安全性とスコープ

このリポジトリは研究プロトタイプです。

次の用途を意図していません。

- 本番環境での自律的意思決定
- 教師なしの実世界制御
- 法務、医療、金融、規制に関する助言
- 実在の個人データまたは機密運用データの処理
- 完全または普遍的な安全性カバレッジの実証
- 外部への自動提出、アップロード、送信、デプロイ
- 外部副作用に対する人間レビューのバイパス

各例は、次のものとして読むべきです。

- ローカル研究シミュレーション
- 教育用リファレンス
- ガバナンスと安全性のテストベンチ
- fail-closed 設計例
- 人間レビュー・ワークフロー例
- 監査ログとチェックポイント整合性の実験
- ローカル・バッチオーケストレーション例

外部提出、アップロード、プッシュ、送信、デプロイ、削除、または実世界制御アクションは、ブロックされるか、人間レビューでゲートされるべきです。

## 責任ある利用と禁止される利用

レビュー・ワークフローとシミュレータ例は、ユーザーが所有するリポジトリ、または明示的な許可を得たリポジトリの防御的レビューを目的としています。

このリポジトリまたはツールを、次の目的に使用しないでください。

- 無許可の脆弱性探索
- 攻撃的な偵察
- 第三者システムまたはコードの悪用
- 認証情報、トークン、シークレットの収集
- 許可のないリポジトリまたはシステムのスキャン
- 実在ターゲットに対する悪用手順の生成または検証
- アクセス制御、レート制限、セキュリティ境界の回避
- 責任ある開示なしに機密性の高い検出結果を公開すること

検出結果は、安全性、信頼性、追跡可能性、ガバナンスの改善に使うべきです。第三者またはオープンソースコードをレビューする場合は、プロジェクトメンテナのセキュリティポリシーと責任ある開示プロセスに従ってください。

## 助言的レビュー・ワークフロー

Tasukeru Analysis は、このリポジトリにおける防御的リポジトリレビュー用の助言的レビュー・ワークフローです。

これは、マルチエージェントおよびエージェント型ワークフローが複雑化するにつれて、その振る舞いが検査、説明、監査しにくくなる問題を点検・統治するためのものです。

次のようなレビュー支援の問いに焦点を当てます。

- 検出結果が適切なレビュー水準に分類されているか
- プロセスログと生成出力の一貫性が保たれているか
- 不確実または危険なケースが人間レビューへ戻されるべきか
- 公開PRコメントが過度なセキュリティ詳細を避けているか
- ワークフローが自動修正、コミット、プッシュ、マージなしの助言のみの状態を維持しているか

このワークフローは詳細な悪用手順を開示せず、自律的な強制システムとして振る舞いません。詳細な検出結果は、人間レビュー用のワークフロー成果物およびログに残ります。

次のような静的チェックおよびリポジトリ固有チェックを実行します。

- Ruff
- Bandit
- pip-audit
- リポジトリ固有のロジックチェック
- ドキュメントレビュー
- 人間レビュー分類

このワークフローは、すべての助言的検出結果を自動的にブロッキング失敗として扱うのではなく、保守性、安全レビュー、開発者の使いやすさを改善するよう設計されています。

警告を隠すことを目的としていません。代わりに、実際の修復候補、レビューのみの検出結果、履歴バージョン警告、ノイズらしきものを分離し、メンテナが重要な項目に集中できるようにします。

## 助言のみポリシー

助言的レビュー・ワークフローは、次を行いません。

- ブランチを自動作成する
- Pull Request を自動作成する
- 変更を自動コミットする
- 変更を自動プッシュする
- 修正を自動適用する
- Pull Request を自動マージする

人間レビューを必要とする検出結果だけが、PRコメントを生成することがあります。

その他の検出結果は、人間レビュー用に GitHub Step Summary とワークフロー成果物へ収集されます。

このワークフローは、人間レビューの負荷を下げることを目的としており、人間の意思決定を置き換えるものではありません。

## 人間が読めることに関するポリシー

このプロジェクトは、人間が安全に読み、レビューし、保守できるコードを優先します。

AIや自動ツールが処理しやすいという理由だけで、変更を良い修正として扱うべきではありません。変更が人間にとっての可読性、レビュー容易性、保守性を低下させる可能性がある場合は、人間レビューに戻すべきです。

## セキュリティレビュー出力ポリシー

セキュリティ関連の出力は、防御的レビューと安全な修復方針に焦点を当てるべきです。

公開PRコメントには、次を含めるべきではありません。

- 悪用ペイロード
- 攻撃コマンド
- 攻撃的な再現手順
- 段階的な悪用手順
- 第三者ターゲットの悪用詳細

セキュリティ関連の検出結果は、次に焦点を当てるべきです。

- 影響を受けるファイルと行
- レビューが必要な防御上の理由
- 期待される安全上の影響
- 安全な修復方向
- 人間レビューが必要かどうか

## 人間レビューモデル

このリポジトリは、対応が必要な問題と想定されるノイズを分離するためにレビュー水準を使います。

現在のレビュー水準は次のとおりです。

- `HITL_REQUIRED`: マージまたはさらなるアクションの前に人間レビューが必要
- `FIX_RECOMMENDED`: 具体的な修正が適切である可能性が高い
- `REVIEW_RECOMMENDED`: 判断前に文脈レビューが必要
- `NOISE_CANDIDATE`: テスト、デモ、研究シミュレーションでは許容される可能性が高い
- `INFO_ONLY`: 履歴バージョン警告を含む情報提供のみ

この分類は既定で助言的です。メンテナが修正、文書化、抑制、受け入れのどれを選ぶかを判断する支援として使われます。

## アクティブレビューキュー

メインのレビューキューは、コンパクトに保つことを意図しています。

即時対応が必要な検出結果は、次として現れるべきです。

- `HITL_REQUIRED`
- `FIX_RECOMMENDED`
- `REVIEW_RECOMMENDED`

これらの件数がゼロの場合、現在のワークフローが人間による修理またはレビューの優先対象として推奨するアクティブな助言項目はありません。

これはリポジトリに警告が存在しないという意味ではありません。残りの警告が、ノイズ候補または履歴・情報記録として分類されていることを意味します。

## 履歴バージョン警告

履歴バージョン警告は完全には除外されません。

助言的ワークフローは、置き換え済みまたは履歴的なシミュレータファイルに由来する検出結果を、監査上有用だがアクティブな修復対象ではない場合に `INFO_ONLY` として可視化します。

これにより、現在人間の注意を必要とする検出結果にアクティブレビューキューを集中させつつ、次の目的で保持されている古いバージョンファイルの可視性を維持します。

- 再現性
- 回帰比較
- 履歴参照
- バージョン付き実験
- 設計比較

履歴ファイルが再びアクティブなエントリーポイントになる場合、その警告は再検討すべきです。

## 典型的な分類例

例:

### subprocess 実行

通常は `HITL_REQUIRED`。

subprocess 実行は外部副作用を生み得るため、レビューが必要です。

### subprocess import

通常は `REVIEW_RECOMMENDED`。

subprocess を import すること自体は外部副作用ではありません。

### テスト内の assert 使用

通常は `NOISE_CANDIDATE`。

pytest テストでは `assert` が一般的に使われます。

### アクティブな実行時コード内の assert 使用

通常は `FIX_RECOMMENDED`。

実行時の `assert` 文は Python の最適化フラグによって削除される可能性があります。

### 置き換え済み履歴シミュレータファイル内の assert 使用

通常は `INFO_ONLY`。

完全除外ではなく、履歴バージョン警告として可視化されます。そのファイルが再びアクティブなエントリーポイントにならない限り、即時修正は不要です。

### テスト内の一時パス使用

`tmp_path` のような分離された pytest fixture を使う場合、通常は `NOISE_CANDIDATE`。

ハードコードされた共有パスを使う場合はレビューすべきです。

### アクティブシミュレーション内の疑似乱数生成器使用

通常は `REVIEW_RECOMMENDED`。

決定的なシミュレーション用乱数は、シークレット、トークン、認証、認可、暗号用途でなければ許容される場合があります。

受け入れる場合、その乱数がシミュレーション専用であることをコードに文書化すべきです。

### 置き換え済み履歴シミュレータファイル内の疑似乱数生成器使用

通常は `INFO_ONLY`。

履歴バージョン警告として可視化されます。セキュリティ上重要な振る舞いまたはアクティブな実行時振る舞いに再利用されない限り、即時修正は不要です。

### 依存関係監査ツールによる依存関係脆弱性

通常は `FIX_RECOMMENDED`。

## 証拠、説明、予測

助言的ワークフローは、検出結果に構造化された意思決定支援メタデータを付与する場合があります。

これには次が含まれます。

- `evidence`: ソースツール、ルールID、ファイル、行、スニペット、成果物参照
- `explanation`: その検出結果が重要な理由と、修復方向が提案される理由
- `fix_prediction`: 提案変更により期待される安全性または保守性上の効果
- `fix_verification`: 候補検証のプレースホルダーまたは結果データ

`fix_prediction` は保証ではありません。レビュー支援情報です。

`fix_verification` が有効な場合でも、助言のみでなければなりません。候補チェックは一時ファイルまたは分離ファイルを使い、人間が明示的に手動適用を選ばない限り、リポジトリファイルを変更してはなりません。

## 出力成果物

助言的ワークフローは、次のようなレビュー成果物を生成する場合があります。

- 助言サマリー
- 人間レビュー報告
- 機械可読レビュー・データ
- 通知状態記録
- PRコメント下書き
- ドキュメント提案下書き
- 信頼度レポート
- 結果一貫性レポート
- 依存関係一貫性レポート

これらの成果物はレビュー資料です。リポジトリファイルを変更せず、自動適用されてはなりません。

人間可読レビュー報告には、次が含まれる場合があります。

- ファイルパス
- 行番号
- 検出コード
- 問題サマリー
- レビュー水準
- 推奨アクション

機械可読レポートは、同じ意思決定支援データを構造化形式で提供します。

PRコメント下書きは、人間レビューを必要とし、PR通知の条件を満たす検出結果にのみ使用されます。

ドキュメント提案下書きには、最小差分案、全文下書き、ハッシュ、証拠、レビュー用チェックリストが含まれる場合があります。これらはリポジトリファイルを変更せず、自動適用されてはなりません。

信頼度スコアはレビュー支援用メタデータにすぎません。検出結果を無視して安全であることを示すものではなく、自動修正、自動PR作成、自動コミット、自動プッシュ、自動マージを有効化してはなりません。

結果一貫性レポートは、プロセスログと生成された結果成果物が一致しているかを記録します。

依存関係一貫性レポートは、検出結果、影響構造、影響分類、レビュー水準、生成出力が一貫しているかを記録します。

これらはすべて助言のみの成果物です。検出結果の変更、分類変更、修正適用、コミット作成、変更のプッシュ、Pull Request のマージを自動で行うものではありません。

## 助言的な振る舞い

助言的ワークフローは、メンテナが次を判断する支援をすべきです。

- どの検出結果がノイズらしいか
- どの検出結果をレビューすべきか
- どのファイルに注意が必要か
- なぜその検出結果が関連するのか
- 推奨修正は何か
- マージ前に人間レビューが必要か
- アクティブな修復候補か、履歴バージョン警告か

ワークフローは人間レビューを置き換えません。より安全なリポジトリ保守のためのトリアージと文書化支援です。

## リポジトリの目的

このリポジトリの主目的は、明示的チェック、再現可能なログ、中断点、人間レビューによって、エージェント型ワークフローをどのように制御できるかを探索することです。

プロジェクトは次を重視します。

- fail-closed 動作
- 危険なアクション前の人間レビュー
- 固定された安全チェック順序
- 安定した理由コード
- チェックポイントと再開動作
- 改ざん検知可能な監査記録
- 成果物整合性検証
- 再現可能なシミュレーション出力
- 助言と実行の明確な分離

このリポジトリは、完全なAI安全性カバレッジを提供すると主張しません。オーケストレーションの振る舞いを研究するための研究・教育用テストベンチです。

## アーキテクチャ図

本番志向のドキュメント・オーケストレータ・シミュレータに関する詳細なアーキテクチャ図は、ドキュメントディレクトリにまとめられています。

図ドキュメントには次が含まれる場合があります。

- ドキュメント・オーケストレータ概要

## 現在のシミュレータ系統

このリポジトリには複数のローカル専用シミュレータ系統があります。各系統は異なるオーケストレーションまたはレビュー問題に焦点を当てます。

### 1. ゲートベース・メディエータ・シミュレータ

このシミュレータは単純な流れを研究します。

```text
Agent → Mediator → Orchestrator
```

主要特性:

- エージェントがタスク入力を正規化する
- メディエータは助言のみを行う
- メディエータに実行権限はない
- オーケストレータは固定順序でチェックを評価する
- 曖昧または相対的な要求は人間レビューへルーティングされる
- 高リスクケースはブロックされ得る
- 生テキストは永続化すべきではない
- 監査ログは個人データの直接漏洩を避けるべきである

標準チェック順序:

```text
Meaning → Consistency → Ambiguity Review → External-Action Safety → Dispatch
```

このシミュレータは次を確認するのに有用です。

- 固定されたチェック順序
- 人間レビューへの遷移
- ブロックされる挙動とされない挙動
- メディエータとオーケストレータの分離
- 理由コードの期待値
- 軽量なマルチタスク・オーケストレーション挙動

### 2. ドキュメントタスク・チェックポイント・シミュレータ

このシミュレータは、より強い永続化および整合性制御を備えたドキュメントタスク・オーケストレーションを研究します。

主要特性:

- タスク契約チェック
- 粗いトークン見積もり
- Word / Excel / PowerPoint 形式のタスクシミュレーション
- タスクごとのチェックポイントと再開
- 監査ログ・ハッシュチェーン
- 保護されたチェックポイントファイル
- 成果物整合性記録
- 個人データ安全な監査、チェックポイント、成果物書き込み
- 改ざん検知
- CLIベースのローカルシミュレーション
- 外部への自動提出なし

重要な実装メモ:

このシミュレータは、ドキュメントタスク結果を表すテキストベースの成果物出力を書き込みます。別途ドキュメント生成ライブラリが接続され、テストされない限り、完全に整形された Microsoft Office ドキュメントを生成すると主張しません。

ハッシュとHMACは改ざん検知を提供します。物理的な書き込み防止は提供しません。実際のデプロイでは、HMACキーは環境変数または保護されたキーファイルから供給される必要があり、リポジトリにコミットしてはなりません。

### 8. Officeタスク調停シミュレーション

このシミュレータは、Word / Excel / PowerPoint 形式の合成成果物を使う、ローカル専用のOfficeタスク調停フローをモデル化します。

生成された文書、スプレッドシート、プレゼンテーション出力が、元のユーザータスク指示と整合しているかを確認します。診断ヘルパーはログを読み、異常を検出し、リスク資料のみを出力します。メディエータはマスク済みメタデータパケットのみを受け取り、元タスクと生成出力の違いを照合します。コーディネータは自分で判断せず、ユーザーが選択したエージェントにのみタスクを配布し、スコアがしきい値を超えた場合にのみ人間レビューを起動し、明示的にユーザーが選んだアクションだけを実行します。

シミュレーションフロー:

```text
ユーザーが対象エージェントを選択
↓
コーディネータがユーザー選択エージェントにのみタスクを配布
↓
Word / Excel / PowerPoint 形式の合成成果物が生成される
↓
診断ヘルパーが内部ログを読み、マスク済みメタデータパケットを作成
↓
メディエータがマスク済みメタデータのみを受け取り、元タスクスナップショットと出力を比較
↓
メディエータが衝突スコアを計算
↓
リスク推定が助言メタデータとして付与される
↓
score == 0.8 の場合、結果は warning / draft review のまま
↓
score > 0.8 の場合、実行は人間レビューのために一時停止
↓
ユーザーが次のアクションを選択
↓
コーディネータが明示的にユーザーが選んだアクションのみを実行
```

このシミュレータは実Officeファイルを生成しません。合成された Word / Excel / PowerPoint 形式のレコードを使って、一貫性、マスキング、しきい値挙動、人間レビューへのルーティング、下書きのみの修正伝播をテストします。

主要特性:

- 固定された Word / Excel / PowerPoint 合成タスクセット
- ユーザー選択エージェントへの配布
- 元タスクスナップショットとハッシュ固定されたタスク基準
- 診断ログ分析とマスク済みメタデータ・ハンドオフ
- メディエータへの生ログ・ハンドオフなし
- Office出力一貫性チェック
- 利益、数式、チャート、結論の不一致検出
- 個人データおよび機密信号のマスキング
- しきい値ポリシー: `score == 0.8` は warning / draft review のみ
- しきい値ポリシー: `score > 0.8` は人間レビューを起動
- ユーザー範囲内の下書き修正提案のみ
- 関連エージェントは下書き修正提案のみを作成
- コーディネータに自律的判断権限なし
- 監査ログ検証
- 結果一貫性検証
- 必須 Word / Excel / PowerPoint 成果物完全性チェック
- 必須成果物の欠落を検出
- ゼロ値を有効値として扱う
- safety-buffer メタデータが明示的かつレビュー可能
- 内部の生エージェントログはデフォルト保存されず、明示的なシミュレーション専用 opt-in が必要
- 実Office文書生成なし
- 外部APIアクセスなし
- 実プロセス制御なし
- 自動修正、コミット、プッシュ、マージなし

このシミュレータは次を確認するのに有用です。

- Word / Excel / PowerPoint 形式の出力が一貫しているか
- Excel数式結果とPowerPointチャート値が矛盾するか
- Word本文、スプレッドシート値、プレゼン要約が乖離するか
- 元のユーザー指示が比較基準として維持されるか
- 個人データおよび機密信号が調停前にマスクされるか
- メディエータがマスク済みメタデータのみを使うか
- `score == 0.8` が人間レビューを起動しないか
- `score > 0.8` が人間レビューを起動するか
- ユーザー対象の修正プロンプトが下書き提案のみを生成するか
- コーディネータが自己判断を避け、明示的にユーザーが選んだアクションだけを実行するか

### 9. 信頼度からリスクへの自動化下書き拡張

この下書き拡張は、Officeタスク調停系統に自動化開始および自動化継続診断を追加します。

主要特性:

- trust-score ベースの自動化開始診断
- `trust_score == 0.9` は自動化対象ではない

- 自動化開始後は automation-risk ベースの継続診断
- `automation_risk_score < 0.1` は自動化を継続

- 自動化再開にはユーザーレビューが必要
- 診断ヘルパーは診断のみであり、エージェントへ直接命令しない
- 自動修正、コミット、プッシュ、マージなし

この拡張は次を確認するのに有用です。

- 自動化開始が `trust_score > 0.9` によってゲートされるか
- `trust_score == 0.9` が不適格として扱われるか
- 自動化がアクティブになる前にユーザー承認が必要か
- 自動化継続が trust scoring から risk scoring へ切り替わるか
- `automation_risk_score == 0.1` が fail-closed で停止へ入るか
- 停止された自動化の再開にユーザーレビューが必要か
- 自動修正、コミット、プッシュ、マージが無効のままか

### 10.1 Mediator / Maestro ゲート強化下書き

この下書き系統は、ソース根拠付きOfficeオーケストレーション・シミュレーションに、明示的なメディエータからMaestroへのレビュールーティングと fail-closed スコア処理を追加します。

ファイル:

- `agent_source_grounded_office_orchestration_sim_v1_0_0_refactored_integrated_draft.py`
- `agent_source_grounded_office_orchestration_sim_v1_0_1_fail_closed_hardening_draft.py`
- `tests/test_agent_source_grounded_office_orchestration_sim_v1_0_1_fail_closed_hardening_contract.py`

`v1.0.0` ファイルは比較基準として保持されます。  
`v1.0.1` ファイルは、基準版の検証済み数値境界挙動を維持しつつ、無効なしきい値スコア入力に対する fail-closed 処理を追加します。

Mediator / Maestro ゲート判断:

- `HITL_REVIEW_READY`: 結果を人間レビュアーに提示してよいが、自動受理または外部適用はされない。
- `PAUSE_FOR_HITL`: 責任下限が満たされない場合、残り適合しきい値が満たされない場合、またはしきい値入力を安全に評価できない場合にワークフローを一時停止する。
- `ROUTED_TO_PRECHECK`: 重大条件または構造的不変条件違反が検出された場合、通常の候補比較へ入らない。

Fail-closed しきい値挙動:

```text
invalid score input
→ PAUSE_FOR_HITL
```

対応する契約テストは次を対象にします。

- 無効スコア入力が fail-closed で `PAUSE_FOR_HITL` になること
- 基準版の数値しきい値挙動の維持
- 責任下限違反がスコアによって救済されないこと
- 重大条件と構造的不変条件違反が `ROUTED_TO_PRECHECK` へルーティングされること
- 候補提示と次段階承認の分離
- ユーザー受理によって `PAUSE_FOR_HITL` または `ROUTED_TO_PRECHECK` がバイパスされないこと
- Maestro による自律的最終判断または外部アクションがないこと
- Tasukeru、Mediator、Maestro によって不正な `sealed=True` 状態が発行されないこと
- 決定的な `runs=100` および `runs=1000` 分布チェック

これはローカル専用の研究・検証下書きです。  
自動採用、自動外部アクション、自動修正、自動コミット、自動プッシュ、自動マージを許可しません。

ファイル:

- 境界付き分離検証承認記録
- 承認、検証実行、適用、マージ、クローズの明示的分離

- 悪用実行なし
- 外部スキャンなし
- 外部システムアクセスなし
- リポジトリ変更なし
- 自動修復なし
- 自動適用なし
- 自動マージなし
- 本番安全保証なし

- セキュリティレビュー・サブエージェントが読み取り専用の助言境界内に留まるか

公開スコープ境界:

この公開系統は、承認記録、静的候補レビュー、調停、人間レビュールーティング、境界付きの実行されない提案記録の範囲に留まるべきです。攻撃的能力拡張、外部実行、外部スキャン、自動修復、自動適用、自動マージ、自律的セキュリティ強制、実在ターゲットに対する悪用検証はスコープ外です。

## バッチ実行と再開

このリポジトリには、バッチ形式のオーケストレーション例が含まれます。

バッチ実行とは、チェック判断、監査記録、中断点を保持しながら、複数のドキュメント関連タスクを1つのオーケストレーション実行として評価できることを意味します。

### メディエータベースのバッチフロー

メディエータベースのシミュレータは、1回の実行で複数のタスクを受け入れられます。

利用例:

- 複数のスプレッドシートまたはスライドタスクをまとめて評価する
- タスク単位のチェック結果を比較する
- 複数タスクにまたがる人間レビュー挙動を確認する
- オーケストレーション前にメディエータ助言を検証する
- 各タスクが固定チェック順序を通ることを確認する

この系統は、次に焦点を当てる場合に有用です。

- マルチタスクのチェック挙動
- メディエータ分離
- 曖昧性レビューと人間レビュー遷移
- 理由コードの安定性
- 軽量なオーケストレーションテスト

タスクシーケンス例:

- T1: スプレッドシートタスク
- T2: プレゼンテーションタスク
- T3: 人間レビューを必要とする曖昧なタスク

期待される挙動:

- 各タスクがエージェントによって正規化される
- 各タスクがメディエータ助言を受ける
- 各タスクがオーケストレータによって評価される
- 一時停止したタスクは、高リスク安全チェックでブロックされない限り非最終のまま残る
- dispatch は最終判断が continue の場合にのみ行われる

### ドキュメントタスク・バッチフロー

ドキュメントタスク・チェックポイント・シミュレータは、固定されたドキュメントタスクシーケンスを使います。

```text
Word → Excel → PowerPoint
```

この系統は、次に焦点を当てる場合に有用です。

- バッチタスク実行
- タスクごとのチェックポイント
- 中断と再開
- 成果物整合性記録
- 保護されたチェックポイント
- 監査ログ・ハッシュチェーン
- 改ざん検知

タスクが中断された場合、シミュレータは次を記録します。

- 失敗したタスクID
- 失敗したチェック
- 理由コード
- チェックポイントパス
- 再開要件
- 該当する場合の人間確認要件

後続の実行は、必要な場合に人間確認後、チェックポイントから再開できます。

## バッチスクリプト

バッチスクリプトは、ローカルシミュレーション実行の便利なラッパーとして使われる場合があります。

推奨されるスクリプト例:

- ローカルデモ実行
- チェックポイント再開実行
- 改ざん検知チェック実行

これらのスクリプトは、ローカル開発者ユーティリティとしてのみ扱うべきです。

次を行うべきではありません。

- 本番HMACキーを埋め込む
- 成果物を自動アップロードまたは自動提出する
- 人間確認をバイパスする
- ファイルを自動削除する
- 実在の個人データまたは機密データを処理する
- ライセンスまたは安全性の意味を変更する
- テストまたは安全チェック不変条件を弱める

ローカルデモでは、シミュレータが明示的に対応している場合のみデモキー・モードを使用できます。本番に近い実行では、HMACキーを環境変数または保護されたキーファイルから供給すべきです。

### バッチスクリプトの役割例

#### ローカルデモスクリプト

目的:

- ローカルデモを実行する
- 安全なサンプル入力を使う
- 監査ログとシミュレートされた成果物をローカル出力ディレクトリへ書き込む
- 外部提出を避ける

#### 再開スクリプト

目的:

- チェックポイントから再開する
- 明示的な再開確認を要求する
- 継続前に完了済み成果物を検証する
- チェックポイント整合性を保持する

#### 改ざんチェック・スクリプト

目的:

- 監査ログ、チェックポイント、成果物の整合性挙動を検証する
- 改ざん検知を実演する
- 整合性検証に失敗した場合、人間レビューのために一時停止する

## ゲートと判断モデル

このリポジトリは、オーケストレーション挙動を追跡可能にするため、明示的なゲート判断を使います。

一般的な判断には次が含まれます。

- `RUN`
- `PAUSE_FOR_HITL`
- `STOPPED`

一般的な解釈:

- `RUN`: シミュレータは次のチェックまたはディスパッチステップへ進んでよい
- `PAUSE_FOR_HITL`: シミュレータは一時停止し、人間レビューを待つべき
- `STOPPED`: シミュレータはブロッキング条件に到達した

曖昧または相対的な要求は、安全と黙示的に扱うのではなく、人間レビューのために一時停止すべきです。

高リスク条件、特に外部副作用、ポリシー違反、安全でないアクションは、シミュレータ契約に応じてブロックされるか、人間レビューへルーティングされるべきです。

## 監査と整合性モデル

監査および整合性挙動は、シミュレータ系統によって異なります。

メディエータベースのシミュレータは、軽量な監査イベントと安全なコンテキストログに焦点を当てます。

ドキュメントタスク・チェックポイント・シミュレータは、より強い整合性制御を追加します。

- 監査ログ・ハッシュチェーン
- 行単位HMAC
- チェックポイントHMAC
- 成果物SHA-256
- 成果物HMAC
- 再開時の完了済み成果物検証
- 改ざん検知

これらの仕組みは変更を検出可能にすることを目的としています。ローカル主体がディスク上のファイルを変更することを防止するものではありません。

## チェックポイントと再開モデル

チェックポイントベースのシミュレータは、中断された実行をサポートするよう設計されています。

チェックポイントは次を記録する場合があります。

- run ID
- current task ID
- failed task ID
- failed check
- reason code
- task status
- artifact path
- artifact hash
- resume が許可されるか
- resume 前に人間レビューが必要か

再開時には、シミュレータが継続する前に完了済み成果物を検証すべきです。検証に失敗した場合、黙って継続するのではなく、人間レビューのために一時停止すべきです。

## 成果物モデル

シミュレータの成果物出力は、研究成果物として扱うべきです。

シミュレータ系統によって、出力には次が含まれる場合があります。

- 成果物プレビュー
- テキストベースのドキュメントタスク出力
- 監査JSONLファイル
- チェックポイントJSONファイル
- 整合性メタデータ
- サマリー記録

実際のドキュメント生成コードが追加されテストされない限り、リポジトリはシミュレートされた出力を完全な本番Officeドキュメントとして説明すべきではありません。

外部副作用には次のようなアクションが含まれます。

- メール送信
- ファイルアップロード
- 成果物提出

- ファイル削除
- 外部API呼び出し
